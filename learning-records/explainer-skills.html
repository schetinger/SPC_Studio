Além do Grill-Me: Manutenção, Debug e Refatoração

É excelente que você já domine o fluxo de criação de features (/grill-me → /to-issues → /tdd). Esse é o coração do desenvolvimento guiado por agentes!

Agora, vamos expandir sua Zone of Proximal Development para lidar com código existente, bugs e melhoria contínua.

💡 Existe uma skill específica para "Debug"?
Curiosamente, não existe uma skill /debug isolada! O motivo é que debugar com agentes funciona melhor quando combinamos a investigação conversacional com o TDD. Se você encontrar um bug, o fluxo ideal é:
  1. Explicar o bug para o agente na conversa normal (fornecendo logs ou descrevendo o problema).
  2. Pedir para o agente investigar a causa raiz.
  3. Chamar a skill /tdd para escrever um teste que falhe reproduzindo o bug, e então consertá-lo.

Skills para Expandir seu Arsenal

🔍 /review

Para que serve: Revisa mudanças recentes (um branch, PR ou alterações não commitadas) garantindo que o código segue as diretrizes do projeto e atende aos requisitos da Spec original.

Quando usar para Debug: Se você implementou algo e começou a dar erro, rodar um /review pode fazer o agente encontrar inconsistências lógicas ou violações de padrões de forma estruturada antes mesmo de você começar a caçar o bug manualmente.

🚀 Try this:

Faça uma pequena alteração intencional no código (como mudar o nome de uma variável ou apagar uma validação em models.py) e diga ao agente: "Invoque a skill /review nas alterações recentes."

🏗️ /improve-codebase-architecture

Para que serve: Encontra oportunidades de melhoria estrutural (refatoração) baseando-se no vocabulário do domínio (CONTEXT.md / AGENTS.md / GLOSSARY.md) e em decisões de arquitetura documentadas.

Quando usar para Debug: Bugs frequentes geralmente nascem de arquiteturas confusas. Se um módulo (ex: consumers.py) está muito grande e complexo, invocar essa skill vai quebrar o problema em partes menores, mais fáceis de testar e debugar.

🚀 Try this:

Selecione um arquivo complexo do Django (como views.py ou services.py) e diga: "Use /improve-codebase-architecture para analisar este serviço."

🗺️ /zoom-out

Para que serve: Pede para o agente se afastar do detalhe do código e mapear as relações entre os módulos usando a linguagem de negócios do projeto.

Quando usar para Debug: Quando você não tem ideia de onde o bug está vindo, ou quando herda um código que você não conhece. O mapa te ajuda a descobrir qual camada está falhando (ex: é no WebSocket? no ORM? no frontend?).

Outras Skills Úteis para Experimentação

  • /prototype: Cria um app descartável (terminal ou web) para testar uma ideia rapidamente antes de combar com o /to-issues. Útil para testar lógica de estado sem sujar o projeto.
  • /grill-with-docs: Parecido com o /grill-me, mas foca em alinhar a sua ideia com a documentação existente do domínio, atualizando os arquivos CONTEXT.md ou ADRs simultaneamente.
