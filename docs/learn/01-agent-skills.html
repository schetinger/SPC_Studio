1 O que são agent skills?

Uma skill é um arquivo de instruções (SKILL.md) que ensina um agente de IA a executar um workflow especializado de engenharia de software — como um manual de procedimento operacional padrão.

Quando você invoca uma skill (digitando /to-issues, por exemplo), o agente lê o SKILL.md correspondente e segue as instruções ao pé da letra. Você não precisa explicar como fazer — só precisa fornecer o quê.

💡
A analogia certa

Pense como contratar um engenheiro sênior com um manual de práticas da empresa. Você descreve o problema, ele segue o processo. As skills são esse manual.

2 O mapa mental: quando usar cada skill?

A regra de ouro é: spec antes de código. O fluxo natural é linear.

1

Você tem uma ideia ou backlog

Ex: "Quero adicionar exportação CSV das cartas de controle do CEP."

2

Especifica com /to-issues

O agente quebra a ideia em issues rastreáveis no GitHub, com critérios de aceite mensuráveis. Cada issue fica ready-for-agent.

📋 /to-issues Adicionar exportação CSV para cartas Xr e IMR
3

Implementa com /tdd

Aponta para a issue criada. O agente escreve testes primeiro (red), depois implementa (green), depois limpa (refactor).

🔴 /tdd issue #12
4

Revisa com /review

Audita as mudanças em dois eixos: padrões de código do repo e fidelidade à spec original da issue.

🔍 /review since main
5

Mergeia e repete

Próxima issue da lista. O ciclo acelera porque cada issue já nasce especificada.

3 /to-issues em detalhes

Esta é a skill mais importante para delegação. Ela transforma uma ideia vaga em issues independentes e auto-suficientes — ou seja, um agente consegue implementá-las sem precisar te perguntar nada.

A skill usa o conceito de tracer-bullet vertical slice: cada issue corta verticalmente todas as camadas (model → view → teste) de uma feature pequena, ao invés de fazer uma issue por camada.

⚠️
Por que isso importa para o CEP?

No seu projeto, uma issue do tipo "criar endpoint de exportação CSV" cortaria: views.py (nova view), urls.py (nova rota), utils.py (lógica de geração CSV) e tests.py — tudo numa fatia vertical testável.

Exemplo real para o projeto CEP:

# Você digita no chat: /to-issues Quero que a API retorne os alertas Western Electric formatados por tipo de regra, com timestamp de quando ocorreram, em vez do formato atual de lista plana.

A skill vai ler seu AGENTS.md, entender que o issue tracker é GitHub, quebrar isso em 2-3 issues com critérios de aceite claros, e criar no GitHub via gh issue create. Cada issue sai com label ready-for-agent.

📎
Citação da skill

to-issues/SKILL.md — "Break a plan, spec, or PRD into independently-grabbable issues on the project issue tracker using tracer-bullet vertical slices."

4 /tdd em detalhes

Implementa features com o loop red → green → refactor, guiado pelo agente, a partir de uma issue existente. Você fornece a issue, o agente escreve os testes, faz passar, e limpa.

R

RED — Testes que falham

O agente lê a spec da issue e escreve testes que descrevem o comportamento esperado. No CEP: test_alertas_western_electric_formato_correto() falharia aqui.

G

GREEN — Implementação mínima

O agente implementa o mínimo necessário para fazer os testes passarem. Sem otimizações, sem over-engineering.

♻

REFACTOR — Limpeza com testes verdes

O agente melhora o código sem alterar comportamento. Os testes continuam passando como garantia.

🧪
Pré-requisito importante

O /tdd funciona melhor quando a issue de origem tem critérios de aceite claros. Por isso a sequência correta é sempre: /to-issues → /tdd. Nunca inverta.

5 /review em detalhes

Audita mudanças de código em dois eixos paralelos, rodados por sub-agentes simultâneos:

📐
Eixo 1: Standards
O código segue os padrões documentados deste repositório? Convenções, estrutura, nomenclatura do glossário.
📋
Eixo 2: Spec
O código implementa o que a issue/PRD pediu? Nada a mais, nada a menos. Fiel à spec original.

Como invocar:

# Revisa tudo desde o branch main: /review since main # Revisa desde um commit específico: /review since abc123f # Revisa apontando para a issue que gerou o trabalho: /review since main spec issue #12
🎯
Para o projeto CEP

O reviewer vai verificar se a sua implementação de alertas Western Electric em models.py está aderente à issue, e se segue os padrões do repo (imports, nomenclatura, estrutura de classes).
